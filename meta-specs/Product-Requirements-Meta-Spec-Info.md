# Product-Requirements-Meta-Spec-Info

> 제품·요구사항 카테고리의 명세 유형 카탈로그.
> 각 유형: **언제 필요한가(트리거) · 정의 · 목차 · 섹션 규칙 · 참조관계.**
> 에이전트는 트리거를 프로젝트에 대입해 작성 여부를 판단한다.

---

## North Star

- **언제 필요한가:** 제품/시스템이 존재하면 거의 항상. 가장 먼저 작성한다.
- **정의:** 이 제품이 왜 존재하는지를 한 문장으로 응축한 최상위 지향점.
- **참조관계:** 모든 PRD 항목의 근거. 다른 명세를 참조하지 않는다 (최상위).
- **ID:** `NS`

**목차 + 섹션 규칙**
1. **한 문장** — 8단어 이내. "누구를 위해 무엇을 가능하게 하는가"가 드러나야 함.
2. **이 문장이 아닌 것 (비목표)** — 흔한 오해나 명시적으로 배제하는 방향 2~3개.
3. **성공의 모습** — 이 별이 달성됐을 때 세상이 어떻게 다른지 2~3줄.

---

## PRD (Product Requirements Document)

- **언제 필요한가:** 사람 사용자나 이해관계자가 있으면 거의 항상. North Star 다음.
- **정의:** 누구를 위해 어떤 문제를 어떻게 해결하는지를 정의하는 제품 요구 문서.
- **참조관계:** North Star를 근거로 함. 각 항목은 1개 이상 SRS로 이어져야 함 (§Master 4).
- **ID:** User Story는 `PRD-US-NN`

**목차 + 섹션 규칙**
1. **배경 / 문제** — 왜 이걸 만드는가. 현재의 고통(pain)을 구체적으로.
2. **대상 사용자** — 누구인가. 페르소나/세그먼트. 막연한 "모두" 금지.
3. **사용자 시나리오 (User Stories)** — 각 항목 `PRD-US-NN`. "[사용자]로서 [목적]을 위해 [행동]한다" 형식.
4. **성공 지표** — 측정 가능한 수치 1개 이상 필수 (예: "주간 생성 회차 ≥ N").
5. **비목표 (Non-goals)** — 이번에 안 하는 것. 범위 폭주 방지.
6. **가정 / 의존** — 전제하는 외부 조건.

---

## SRS (Software Requirements Specification)

- **언제 필요한가:** 구현해야 할 기능이 있으면 거의 항상. PRD를 검증 가능한 단위로 분해할 때.
- **정의:** PRD의 시나리오를 구현·검증 가능한 개별 요구사항으로 분해한 명세 (IEEE 29148 계열).
- **참조관계:** PRD를 근거로 함. 각 SRS-F는 정확히 1개 모듈에 할당됨 (Module Map). NFR은 SRS-N으로.
- **ID:** 기능 `SRS-F-NNN`, 비기능 `SRS-N-NNN`

**목차 + 섹션 규칙**
1. **범위** — 이 SRS가 다루는 경계.
2. **기능 요구사항** — 각 항목 `SRS-F-NNN`. 다음 필드 필수:
   - `maps_to_prd`: 근거 PRD-US (1개 이상). 없으면 고아 — 금지.
   - `owner_module`: 책임 모듈. 미할당 금지.
   - `acceptance`: 수용 기준 (체크 가능한 형태).
3. **비기능 요구사항** — 각 항목 `SRS-N-NNN`. 성능/보안/가용성 등. 측정 가능해야 함 (상세는 Constraints-Operation의 NFR).
4. **추적 매트릭스** — PRD-US ↔ SRS-F ↔ Module 대응표.

---

## User Story / Use Case

- **언제 필요한가:** 사용자 상호작용 시나리오가 있으면. PRD 안에 포함하거나 복잡하면 별도 문서로 분리.
- **정의:** 사용자가 시스템과 상호작용하여 목적을 달성하는 흐름.
- **참조관계:** PRD-US를 상세화. 관련 SRS-F로 이어짐.
- **ID:** PRD-US-NN을 따르거나 `UC-NN`

**목차 + 섹션 규칙**
1. **액터** — 누가 수행하는가 (사람/에이전트/시스템).
2. **사전 조건** — 시작 전 충족 상태.
3. **주 흐름** — 정상 경로 단계별.
4. **대안 / 예외 흐름** — 실패·분기 경로.
5. **사후 조건** — 완료 후 상태.

---

## Acceptance Criteria

- **언제 필요한가:** 완료를 객관적으로 판정해야 할 때. SRS-F마다 권장.
- **정의:** 요구사항이 "완료"되었음을 판정하는 검증 가능한 기준.
- **참조관계:** SRS-F에 종속. Test Plan의 근거가 됨.
- **ID:** 해당 SRS-F-NNN에 귀속

**목차 + 섹션 규칙**
1. **기준 목록** — Given/When/Then 또는 체크리스트 형식. 각 항목 객관적으로 판정 가능해야 함.
2. **경계 / 엣지 케이스** — 빈 값, 한계값, 동시성 등.
3. **명시적 제외** — 이 기준이 다루지 않는 것.
