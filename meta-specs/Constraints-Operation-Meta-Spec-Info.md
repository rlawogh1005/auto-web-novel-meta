# Constraints-Operation-Meta-Spec-Info

> 제약·운영 카테고리의 명세 유형 카탈로그.
> 각 유형: **언제 필요한가(트리거) · 정의 · 목차 · 섹션 규칙 · 참조관계.**

---

## Constraints

- **언제 필요한가:** 기술/법/비즈니스 제약이 있으면 거의 항상. 초기에 작성해 ADR·아키텍처를 제약.
- **정의:** 설계와 구현을 제약하는 고정 조건들 (선택이 아니라 주어진 것).
- **참조관계:** ADR이 이를 인용해 결정을 정당화. PRD/SRS의 가능 범위를 한정.
- **ID:** `CON-NNN`

**목차 + 섹션 규칙**
1. **기술 제약** — 강제되는 스택/플랫폼/버전 (예: "LLM은 Anthropic Claude만").
2. **법적 / 규제 제약** — 개인정보, 저작권, 연령 등.
3. **비즈니스 제약** — 예산, 일정, 인력, 외부 계약.
4. **운영 제약** — 인프라 한계 (예: API rate limit, 단일 머신).
5. 각 제약은 **출처**(왜 이게 제약인지)를 명시.

---

## NFR (Non-Functional Requirements)

- **언제 필요한가:** 성능·보안·가용성 등 품질 요구가 있으면. SRS의 비기능 부분을 상세화.
- **정의:** 기능이 "어떻게" 동작해야 하는지의 품질 기준. 측정 가능해야 함.
- **참조관계:** SRS-N으로 등재. Test Plan이 검증. 일부는 Constraints에서 파생.
- **ID:** `SRS-N-NNN` (SRS와 공유)

**목차 + 섹션 규칙**
1. **성능** — 응답시간, 처리량 목표. 수치 필수 (예: "p95 < 500ms").
2. **확장성** — 부하 증가 시 동작 (예: "동시 N 인스턴스").
3. **가용성 / 신뢰성** — 가동률 목표, 장애 허용.
4. **보안** — 인증/인가/암호화 요구 (상세 위협은 Threat Model).
5. **유지보수성 / 관측성** — 로깅, 모니터링, 추적 요구.
6. 각 항목은 **측정 방법**을 명시 (측정 불가능한 NFR은 금지).

---

## Threat Model

- **언제 필요한가:** 인증·결제·민감정보·외부 입력을 다루면. 보안 표면이 있을 때.
- **정의:** 시스템의 위협을 식별하고 완화책을 정의 (STRIDE 등).
- **참조관계:** NFR(보안)을 구체화. 완화책은 SRS-F/ADR로 이어질 수 있음.
- **ID:** `THR-NNN`

**목차 + 섹션 규칙**
1. **자산** — 보호 대상 (데이터, 키, 계정).
2. **신뢰 경계** — 어디서 신뢰 수준이 바뀌는가 (외부↔내부).
3. **위협 목록** — 각 위협 `THR-NNN`. 유형(스푸핑/변조/유출/권한상승 등) + 영향.
4. **완화책** — 각 위협에 대응하는 방어. 미완화 위협은 "수용된 위험"으로 명시.
5. **검증** — 완화책을 어떻게 테스트하는가.

## Test Plan / Test Spec

- **언제 필요한가:** 검증 전략을 명문화해야 할 때. 다층 테스트(unit~e2e)가 있으면.
- **정의:** 무엇을 어떤 수준에서 어떻게 검증하는지의 전략.
- **참조관계:** Acceptance Criteria와 NFR을 검증. SRS-F별 커버리지 추적.
- **ID:** `TEST-<범위>`

**목차 + 섹션 규칙**
1. **테스트 수준** — unit / integration / e2e 각각의 범위와 책임.
2. **수준별 도구** — 각 수준에서 쓰는 도구.
3. **커버리지 목표** — 측정 기준과 목표치.
4. **E2E 시나리오** — 사용자 관점 핵심 흐름 (UI→DB까지). Acceptance Criteria 기반.
5. **검증 게이트** — CI에서 무엇이 통과해야 머지 가능한가.
6. **독립성 규칙** — 작성 에이전트와 검증 에이전트의 분리 원칙 (자기 코드 자기 검증 금지).

## Runbook

- **언제 필요한가:** 운영/배포/장애 대응 절차가 필요할 때. 프로덕션 운영 단계.
- **정의:** 운영자가 시스템을 배포·모니터·복구하는 실행 절차.
- **참조관계:** Architecture(배포 구조)와 NFR(가용성)에 기반.
- **ID:** `RUN-<주제>`

**목차 + 섹션 규칙**
1. **배포 절차** — 단계별 배포 명령/순서.
2. **모니터링** — 무엇을 보는가, 정상 지표 범위.
3. **장애 시나리오 + 대응** — 흔한 장애와 복구 절차.
4. **롤백** — 되돌리는 방법.
5. **연락 / 에스컬레이션** — 누구에게 언제.
