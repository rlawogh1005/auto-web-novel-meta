# Master-Meta-Spec-Info

> **이 문서는 명세서를 "어떻게 만드는가"를 정의하는 표준의 진입점이다.**
> 에이전트는 명세 작업을 시작할 때 **반드시 이 파일을 먼저 읽는다.**
> 이 파일은 *명세 자체*가 아니라 *명세를 식별하고 작성하는 절차와 규칙*이다.

---

## 0. 이 표준이 존재하는 이유

산업계에는 PRD, SRS, ADR, C4 등 개별 명세 표준은 있으나, **"한 프로젝트에서 어떤 명세를 언제 골라서, 어떤 순서로, 어떤 일관성 규칙으로 작성하는가"**를 통합한 메타 표준은 없다. 그 결과 대부분의 프로젝트는 명세를 직관(주먹구구)으로 작성한다.

이 메타 명세(meta-specs)는 그 공백을 메운다. 목표는 두 가지다:

1. **식별 가능성** — 에이전트가 프로젝트를 보고 "어떤 명세가 필요한지" 스스로 판단할 수 있게 한다.
2. **일관성** — 작성된 명세들이 서로 모순되지 않고 추적 가능하게 한다.

---

## 1. 핵심 원칙

1. **docs/는 비어서 시작한다.** 사람이 미리 어떤 명세가 필요한지 정하지 않는다. 에이전트가 meta-specs를 읽고 식별해서 생성한다.
2. **필요한 것만 만든다.** 모든 명세 유형을 다 만들지 않는다. 각 유형의 "언제 필요한가(트리거)"에 해당할 때만 만든다.
3. **명세는 검증 가능한 단위 ID를 가진다.** ID 없는 요구사항은 추적 불가능하므로 금지한다 (§5).
4. **명세 간 관계는 명시한다.** 어떤 명세가 어떤 명세를 근거로 하는지 추적한다 (§4).
5. **진행 중 프로젝트에도 삽입 가능하다.** 기존 docs/를 스캔해 "빠진 명세"를 식별하고, 기존 것은 검증·보완한다.

---

## 2. 명세 카테고리 라우팅

명세 유형은 4개 카테고리로 나뉜다. 각 카테고리의 상세(유형별 트리거·목차·규칙)는 해당 파일에 있다.

| 카테고리 | 파일 | 다루는 명세 유형 (요약) |
|---|---|---|
| 제품·요구사항 | `Product-Requirements-Meta-Spec-Info.md` | North Star, PRD, SRS, User Story / Use Case, Acceptance Criteria |
| 아키텍처·설계 | `Architecture-Design-Meta-Spec-Info.md` | Architecture Overview (C4), ADR, Module Map, API Contract, Data Model, Sequence/Flow |
| 제약·운영 | `Constraints-Operation-Meta-Spec-Info.md` | Constraints, NFR, Threat Model, Test Plan, Runbook |
| 도메인 | `Domain-Meta-Spec-Info.md` | Domain Model, Bounded Context, Glossary |

---

## 3. 에이전트 작업 절차

명세 작업 지시를 받으면 **이 순서대로** 진행한다.

### Step 1 — 프로젝트 성격 파악
- `../WORLD.md` (있으면) 읽어 세계관·서비스 구성 파악.
- `../docs/` 스캔 — 기존 명세가 있는지 확인. 있으면 "진행 중 프로젝트", 비어 있으면 "신규 프로젝트".
- `../docs/Navigator.md` 읽어 현재 명세 작성 현황 파악.

### Step 2 — 필요한 명세 식별
- 4개 카테고리 메타 파일의 각 명세 유형 **"언제 필요한가(트리거)"**를 이 프로젝트에 대입.
- 트리거에 해당하는 유형만 "작성 대상"으로 선정.
- §6 선택 가이드 표를 빠른 참조로 활용.
- **식별 결과를 먼저 사람에게 보고하고 승인받는다.** 임의로 전부 작성하지 않는다.

### Step 3 — 명세 작성
- 선정된 각 명세를 해당 메타 파일의 **목차 + 섹션 규칙**에 따라 `../docs/`에 작성.
- 작성 순서는 §4 관계도를 따른다 (근거가 되는 명세를 먼저).
- 모든 항목에 §5 ID 규칙 적용.

### Step 4 — 일관성 검증
- §4 참조 규칙을 위반하지 않는지 확인 (고아 항목, 순환 의존, 미근거 결정 등).
- 위반 발견 시 수정 후 진행.

### Step 5 — Navigator 갱신
- `../docs/Navigator.md`에 작성/수정된 명세를 인덱스로 기록 (유형, ID 범위, 상태, 갱신일).

## 4. 명세 간 관계 / 참조 규칙

명세는 계층이 아니라 **방향 있는 참조 그래프**다. 작성 시 근거가 되는 명세를 먼저 만든다.

```
North Star
   └─근거→ PRD ─근거→ SRS ─할당→ Module Map
                  │           │
                  │           └─구현→ API Contract / Data Model
                  ├─제약→ Constraints / NFR
                  └─보안범위→ Threat Model
Domain Model ─용어→ Glossary
ADR ─결정정당화→ (Constraints | SRS 인용)
Architecture ─결정반영→ ADR
```

**참조 규칙 (검증 대상):**
- 모든 SRS 항목은 1개 이상의 PRD 항목을 근거로 한다 (고아 SRS 금지).
- 모든 PRD 항목은 1개 이상의 SRS로 이어진다 (미구현 PRD 경고).
- 모든 SRS-F(기능 요구)는 정확히 1개 모듈이 책임진다 (미할당 금지).
- Module Map의 의존 그래프는 DAG여야 한다 (순환 의존 금지).
- 모든 ADR은 Constraints 또는 SRS를 인용해 결정을 정당화한다.
- 모든 API Contract는 구현 모듈이 존재한다.

---

## 5. ID 규칙

| 명세 유형 | ID 형식 | 예 |
|---|---|---|
| North Star | `NS` (단일) | `NS` |
| PRD User Story | `PRD-US-NN` | `PRD-US-03` |
| SRS 기능 요구 | `SRS-F-NNN` | `SRS-F-007` |
| SRS 비기능 요구 | `SRS-N-NNN` | `SRS-N-002` |
| ADR | `ADR-NNN` | `ADR-012` |
| Module | `MOD-<이름>` | `MOD-AUTH` |
| API Contract | `API-<이름>` | `API-PaymentSvc` |
| Threat | `THR-NNN` | `THR-004` |

- ID는 **불변**이다. 한 번 부여하면 재사용하지 않는다 (삭제돼도 번호를 비워둔다).
- 명세 파일 상단 frontmatter 또는 각 항목 헤더에 ID를 명시한다.

## 6. 명세 유형 선택 가이드 (트리거 빠른 참조)

| 명세 유형 | 언제 필요한가 (요약) | 거의 항상? |
|---|---|---|
| North Star | 제품/시스템이 존재하면 | ✅ |
| PRD | 사람 사용자나 이해관계자가 있으면 | ✅ |
| SRS | 구현해야 할 기능이 있으면 | ✅ |
| User Story / Use Case | 사용자 상호작용 시나리오가 있으면 | 대체로 |
| Acceptance Criteria | 완료를 객관적으로 판정해야 하면 | 대체로 |
| Architecture (C4) | 2개 이상 컴포넌트/서비스가 협력하면 | 대체로 |
| ADR | 되돌리기 어려운 기술 결정을 내릴 때 | 결정 시 |
| Module Map | 코드를 모듈로 분해할 때 | 다중 모듈 시 |
| API Contract | 모듈/서비스가 인터페이스로 통신하면 | 통신 시 |
| Data Model | 영속 데이터(DB)가 있으면 | 데이터 시 |
| Sequence / Flow | 다단계 상호작용 흐름이 복잡하면 | 복잡 시 |
| Domain Model | 도메인 개념이 풍부하면 (DDD) | 도메인 복잡 시 |
| Bounded Context | 도메인을 경계로 나눌 때 | 대형 도메인 |
| Glossary | 팀/에이전트가 용어를 공유해야 하면 | 대체로 |
| Constraints | 기술/법/비즈니스 제약이 있으면 | ✅ |
| NFR | 성능·보안·가용성 요구가 있으면 | 대체로 |
| Threat Model | 인증·결제·민감정보를 다루면 | 보안 시 |
| Test Plan | 검증 전략을 명문화해야 하면 | 대체로 |
| Runbook | 운영/장애 대응 절차가 필요하면 | 운영 시 |

> 상세 트리거·목차·작성 규칙은 각 카테고리 파일을 참조한다.

---

## 7. 신규 vs 진행 중 프로젝트

**신규 (docs/ 비어 있음):**
- Step 2에서 트리거에 맞는 명세를 처음부터 식별·작성.
- North Star → PRD → SRS 순으로 상위부터.

**진행 중 (docs/ 일부 존재):**
- 기존 명세를 §4 참조 규칙으로 검증 (고아/순환/미근거 탐지).
- 트리거에 해당하나 **누락된** 명세만 식별해 추가 작성.
- 기존 명세를 함부로 재작성하지 않는다 — 보완·검증 우선, 변경은 사람 승인 후.
