# Domain-Meta-Spec-Info

> 도메인 카테고리의 명세 유형 카탈로그.
> 각 유형: **언제 필요한가(트리거) · 정의 · 목차 · 섹션 규칙 · 참조관계.**

---

## Domain Model

- **언제 필요한가:** 도메인 개념이 풍부하고 비즈니스 규칙이 복잡할 때 (DDD 적용).
- **정의:** 도메인의 핵심 개념(엔티티/값객체)과 그들 간 관계·규칙을 표현한 모델 (Eric Evans).
- **참조관계:** Data Model의 근거. Glossary의 용어가 여기서 정의됨. PRD의 문제 공간을 모델링.
- **ID:** 개념명 (Glossary·Data Model과 일치)

**목차 + 섹션 규칙**
1. **핵심 엔티티** — 고유 식별자를 가진 개념. 속성과 생명주기.
2. **값 객체 (Value Objects)** — 식별자 없이 값으로 동등한 개념.
3. **집합체 (Aggregates)** — 일관성 경계로 묶이는 엔티티 그룹과 루트.
4. **도메인 규칙 / 불변식** — 항상 참이어야 하는 비즈니스 규칙.
5. **도메인 이벤트** — 도메인에서 의미 있는 사건 (있으면).

---

## Bounded Context

- **언제 필요한가:** 도메인이 커서 하나의 모델로 표현하면 충돌이 생길 때. 대형 도메인을 경계로 분할.
- **정의:** 특정 모델이 일관되게 유효한 경계. 경계마다 같은 용어가 다른 의미를 가질 수 있음 (DDD).
- **참조관계:** Module Map/Architecture의 서비스 경계와 정렬. Domain Model을 컨텍스트별로 분할.
- **ID:** `CTX-<이름>`

**목차 + 섹션 규칙**
1. **컨텍스트 목록** — 각 컨텍스트 `CTX-*`와 책임 범위.
2. **컨텍스트 맵** — 컨텍스트 간 관계 (공유 커널, 고객-공급자, 부패방지층 등).
3. **용어 차이** — 같은 단어가 컨텍스트마다 다른 의미를 갖는 경우 명시.
4. **경계와 서비스 정렬** — 각 컨텍스트가 어느 서비스/모듈에 대응하는가.

## Glossary

- **언제 필요한가:** 팀/에이전트가 용어를 공유해야 할 때. 거의 모든 프로젝트에 권장.
- **정의:** 프로젝트에서 쓰는 용어의 통일된 정의 사전 (유비쿼터스 언어).
- **참조관계:** Domain Model의 개념을 평이하게 정의. 모든 명세가 이 용어를 일관되게 사용.
- **ID:** 용어 자체

**목차 + 섹션 규칙**
1. **용어 목록** — 가나다/알파벳 순. 각 항목: 용어, 정의(1~2문장), 동의어/금지어.
2. **약어** — 프로젝트 약어와 풀이.
3. **컨텍스트별 의미** — Bounded Context가 있으면 컨텍스트별 의미 차이 표기.
4. **표기 규칙** — 코드/문서에서의 표기 통일 (예: 영문 식별자 ↔ 한글 표시명).
