# Architecture-Design-Meta-Spec-Info

> 아키텍처·설계 카테고리의 명세 유형 카탈로그.
> 각 유형: **언제 필요한가(트리거) · 정의 · 목차 · 섹션 규칙 · 참조관계.**

---

## Architecture Overview (C4)

- **언제 필요한가:** 2개 이상 컴포넌트/서비스가 협력하면. 시스템 전체 구조를 한눈에 봐야 할 때.
- **정의:** 시스템을 Context → Container → Component 수준으로 점진 상세화한 구조 명세 (Simon Brown C4).
- **참조관계:** SRS를 만족하는 구조여야 함. 구조 결정은 ADR로 정당화.
- **ID:** 컴포넌트는 Module Map의 `MOD-*`와 연결

**목차 + 섹션 규칙**
1. **System Context** — 시스템과 외부 액터/시스템의 관계. 가장 바깥 그림.
2. **Container** — 배포 단위(서비스/앱/DB). 각 컨테이너의 책임과 기술 스택.
3. **Component** — 컨테이너 내부의 주요 구성요소 (필요한 컨테이너만).
4. **핵심 결정 요약** — 주요 구조 결정과 해당 ADR 링크.
5. **다이어그램** — 텍스트(mermaid 등)로 표현. 그림은 코드와 함께 버전 관리.

## ADR (Architecture Decision Record)

- **언제 필요한가:** 되돌리기 어렵거나 비용이 큰 기술 결정을 내릴 때. 결정 1건당 1파일.
- **정의:** 하나의 기술적 결정과 그 맥락·근거·결과를 기록한 문서 (Michael Nygard).
- **참조관계:** Constraints 또는 SRS를 인용해 정당화해야 함 (§Master 4). Architecture에 반영됨.
- **ID:** `ADR-NNN`

**목차 + 섹션 규칙**
1. **제목 / 상태** — 상태는 Proposed | Accepted | Superseded(by ADR-NNN).
2. **맥락 (Context)** — 어떤 상황/문제에서 결정이 필요했나. 인용: 관련 Constraint/SRS ID.
3. **결정 (Decision)** — 무엇을 선택했나. 한 문장으로 명확히.
4. **대안 (Alternatives)** — 검토한 다른 선택지와 탈락 이유.
5. **결과 (Consequences)** — 이 결정의 긍정/부정적 파급. 새로 생기는 제약.

## Module Map

- **언제 필요한가:** 코드를 모듈로 분해할 때. 다중 모듈/서비스 프로젝트.
- **정의:** 시스템을 모듈로 나누고 각 모듈의 책임·경계·의존을 정의 (Parnas 기준).
- **참조관계:** 모든 SRS-F가 정확히 1개 모듈에 할당됨. 의존 그래프는 DAG.
- **ID:** `MOD-<이름>`

**목차 + 섹션 규칙**
1. **모듈 목록** — 각 모듈 `MOD-*`. 다음 필드 필수:
   - `hides_decision`: 이 모듈이 숨기는 설계 결정 (Parnas — 없으면 모듈이 아니라 그냥 폴더).
   - `implements_srs`: 책임지는 SRS-F 목록 (1개 이상).
   - `public_interface`: 외부 공개 인터페이스 (API Contract 링크).
   - `depends_on`: 의존 모듈.
2. **의존 그래프** — DAG. 순환 발견 시 경계 재설계.
3. **모듈 판정 체크리스트** — 새 모듈 추가 시: ① 숨기는 결정이 있나 ② 인터페이스가 명확한가 ③ 한 가지 이유로만 바뀌나 ④ 책임 SRS가 있나. 모두 yes여야 모듈.

## API Contract / Interface Spec

- **언제 필요한가:** 모듈/서비스가 인터페이스로 통신하면. 폴리레포에서 특히 중요.
- **정의:** 모듈 간 통신 인터페이스의 계약 (입력/출력/에러/버전).
- **참조관계:** Module Map의 모듈이 구현. SRS-F의 기능을 노출.
- **ID:** `API-<이름>`

**목차 + 섹션 규칙**
1. **엔드포인트 / 메서드 목록** — 경로, 메서드, 목적.
2. **요청 스키마** — 타입 명시 (TypeScript interface / JSON Schema / Pydantic 등).
3. **응답 스키마** — 성공 응답 타입.
4. **에러 계약** — 에러 코드와 의미. 클라이언트가 처리할 케이스.
5. **버전 / 호환성** — 버전 정책, 하위호환 규칙.
6. **구현 모듈** — 이 계약을 구현하는 `MOD-*`.

## Data Model / Schema

- **언제 필요한가:** 영속 데이터(DB)가 있으면.
- **정의:** 저장되는 데이터의 엔티티·속성·관계·제약.
- **참조관계:** Domain Model의 개념을 구현. SRS-F의 데이터 요구를 충족.
- **ID:** 엔티티명 (Domain Model과 일치)

**목차 + 섹션 규칙**
1. **엔티티 목록** — 각 엔티티의 속성·타입·키.
2. **관계** — FK, 카디널리티 (1:N, N:M).
3. **제약 / 인덱스** — 유니크, 체크 제약, 주요 인덱스.
4. **스키마 분리** — 멀티 서비스면 어느 서비스가 어느 테이블을 쓰는지.
5. **마이그레이션 정책** — 스키마 변경 관리 방식.

## Sequence / Flow

- **언제 필요한가:** 다단계 상호작용 흐름이 복잡해서 글로 설명이 어려울 때.
- **정의:** 컴포넌트/액터 간 시간순 상호작용 흐름.
- **참조관계:** User Story/Use Case를 시각화. 관련 API Contract 호출 표현.
- **ID:** `FLOW-<이름>`

**목차 + 섹션 규칙**
1. **시나리오 명** — 어떤 흐름인가.
2. **참여자** — 액터/컴포넌트.
3. **시퀀스** — 단계별 메시지/호출 (mermaid sequenceDiagram 권장).
4. **분기 / 실패 경로** — 예외 흐름.
